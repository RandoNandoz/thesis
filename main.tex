\begin{abstract}                %% Mandatory -  maximum 350 words
  Software engineers create unit tests to catch analysis or programming errors. Writing unit test cases takes
  effort, in both identifying the unit test input and the unit test oracle which verifies the absence of bugs.
  Unit test generation techniques aim to automate or reduce the effort required to create these unit tests.
  One such technique, unit test carving, which extracts unit tests from integration tests, is implemented by ExploTest, the tool studied in this thesis.
  Using popular software repositories and known bugs, we evaluate ExploTest's effectiveness in exercising the code-under-test (RQ1), when and why ExploTest works well or does not work well on certain code (RQ2, RQ4),
  the amount of speedup the tool offers (RQ3), and the costs to run the tool, both in terms of time and cognitive load (RQ5). We find the tool works well to exercise the code-under-test,
  with 64\% of generated tests correctly exercising behaviour, however, oracle generation is still an issue, with only 9\% of tests being able to uncover bugs. Limited external dependencies,
  serializable parameters and return values, and code from source code analysis tools contribute to ExploTest being able to generate test cases that correctly exercise code and uncover bugs.
  We compare the runtime of the generated tests and system tests and find that the unit test suite generated by the tool is faster in 33\% of cases. A regression analysis is performed and
  we find using the coefficient of determination that there is a strong positive relationship between the speedup and the system test execution time.
  This suggests unit test carving is most effective when the system test is expensive. A preliminary case study by the author demonstrates there is overlap between using
