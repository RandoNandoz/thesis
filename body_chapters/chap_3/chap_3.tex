\section{Evaluation Dataset}

\subsection{Criteria}

Since unit tests are used to catch bugs in software, it is natural to use software bugs to evaluate
ExploTest. Therefore, we collected 22 bugs
across 12 repositories.

In selecting bugs, we had a few criteria inspired by Pythoscope's \cite{wiatkowski2011pythoscope} creator:

\begin{enumerate}
      \item Since Python is a popular and general purpose programming language, we want bugs from various types
            of Python software.
      \item Due to design limitations associated with ExploTest, the program with the bug must support Python 3.13.7.
      \item Each bug must be easy to reproduce with an associated system test.
      \item Each bug must have an associated function that causes the bug --- i.e., the bug must be detectable by
            unit testing a function.
      \item The root cause of the bug must be from the source code itself, not due to platform issues (e.g., a
            bug in the interpreter or operating system).
      \item The bug cannot be so severe that the program does not execute or cause other processes to crash.
\end{enumerate}

\section{Answering the RQs}

\subsection{ExploTest Outcomes}

The unit tests that ExploTest produces can be of different qualities, or it may not produce a unit test at
all due to technical limitations. As discussed in the introduction, unit tests have to exercise the
function-under-test (FUT) and then assess its return value with an oracle which in Python is an assertion.

If ExploTest does produce a unit test, we assess 5 different qualities of the unit tests.
\subsubsection{Test Qualities}
\begin{itemize}
      \item \textbf{Function called}: Does the tool correctly setup and call the required function?
      \item \textbf{Output correct}: When the function-under-test is called by the generated unit test,
            does it have the same output as the function during the system test?
      \item \textbf{Oracle created}: Are any assertions generated in the test?
      \item \textbf{Passes good code}: Do the assertions generated pass when run on the function-under-test in
            the correct version of the codebase?
      \item \textbf{Fails on bad code}: Do the assertions generated fail when run on the function-under-test in
            the buggy version of the codebase?
\end{itemize}

\subsection{Flakiness}
A unit test is considered flaky when it exhibits both passing and failing results on the same version of the
source code without any changes to the
code itself. We run the test 100 times as a rudimentary test for flakiness.

\subsubsection{Preparation}
For each bug, we will create a new Python virtual environment using the \verb|venv| module.


We then install the package and its dependencies as instructed by the documentation of the package, along
with ExploTest. For seaborn, since the bugs
appear as plots, we install \verb|jupyter| and \verb|notebook| to work with plots interactively.

\subsubsection{Finding carve target}
ExploTest uses an \verb|@explore| decorator to mark functions to carve in the source code. By placing the
decorator, the function is transformed to
first perform a static analysis of the function's source code (e.g., to get the function name and
parameter/package information).

To evaluate RQ1, we can split the dataset into two by bug origin: bugs from BugsInPy and bugs from QuAC. Bugs
from BugsInPy come with buggy functions
already known, while the causes of bugs from QuAC are indeterminate. While the bugs from QuAC may contain
fixes in pull requests that identify the
functions that introduce the bug, we refrain from looking at them to help answer RQ5 with a cleaner slate.

\subsubsection{Locating carve targets for bugs from BugsInPy}
The data provided by BugsInPy already identify the individual lines that cause the bug to occur.

For us, the first thing to do is to place the \verb|@explore| at the nearest function to the lines that cause
the bug to occur.
However, these functions might not be good candidates for carving for two reasons: (1) parameters might not
be serializable, as discussed in chapter
2, or (2), the return value of the function will not allow us to generate an oracle that detects the bug.

Oftentimes, human inspection of the source code allows us to determine the presence of unserializable
objects. We can infer these properties from
variable names, for example, variables named \verb|frame| suggest the presence of frame objects which contain
pointers to native code that cannot be serialized.

A similar manual source code analysis works to identify return values not conducive to good test oracles.
We can inspect the source code for standalone \verb|return|'s, which indicate the function returns \verb|None|.
Functions that return \verb|None| typically indicate that the function is used for its effects only.


When we identify a method that is a bad candidate for test carving, we walk up the call graph of the
function (i.e., look at the caller and the
usage of our target function), and place an \verb|@explore| once we determined that the objects are
serializable, or keep looking up.


\subsubsection{Answering RQ1}
In RQ1, we posed: \textit{To what extent does the unit test carving tool we created generate viable unit tests
      when applied to popular, open-source Python repositories?}

\begin{enumerate}
      \item Run the non-buggy version of the code and function and observe its output.

      \item Use the procedure outlined above to place the \verb|@explore| to mark the function-under-test for carving.

      \item Generate the test on the non-buggy version of the code using the system test case:

            Did the tool crash? Were exceptions or errors emitted and what kind?

      \item If a non-empty file was produced, inspect the file and run the test using a debugger to determine
            whether the FUT was reached. Observe the
            output of the function from the generated unit test against the output of the system test. If they
            are the same, we can consider the FUT to
            be exercised.

      \item When running the test, we can observe whether the test passes on the non-buggy code. If it passes, we
            move and apply the buggy commits. If
            the test fails on the source it was generated on, we note it down and skip the next step.

      \item We then run the test, generated on the non-buggy code on the buggy version of the codebase. The test
            should not pass, and if it does, we
            consider the oracle defective.
\end{enumerate}

This procedure should allow us to bucket each created test by the table, and facilitate discussion of failure cases.

\subsubsection{Answering RQ3}
RQ3 asked: \textit{What is the amount of speedup offered by the unit tests created by the tool compared to their
      system test runs?}

In RQ3, we compare the time it takes for a system test to execute against the time it takes for the unit tests generated by
ExploTest to execute.

\textbf{Measuring system test and unit test performance}

We use Python's \verb|time.perf_counter()|, which measures at nanosecond resolution, to measure the time it takes for a system test to execute. Some system tests
may require several steps (e.g., setting up a web server and using \verb|curl| to make requests to the server), and we use
bash and Python scripts to automate them to remove any human delays in manual testing. Call the run time of the system test
$T_\text{sys}$


The unit tests are run directly using \verb|pytest <paths to unit tests>|. When the \verb|pytest| command is invoked to run
unit tests, PyTest performs two passes, the collection pass, which collects all the unit tests and test configuration files in the directory.
It involves traversing all subdirectories, discovering tests, test configuration files, and importing all modules for the discovered tests \cite{pytestdocs-goodpractices}.
This can take a significant amount of time, often longer than the second pass, which actually executes the unit test code collected from the
collection pass \cite{pytestdocs-importmechanism}.

However, ExploTest arbitrarily chooses the PyTest library to generate its tests, therefore, to get a more accurate look at the speedup
present, without the overhead of test discovery (the other popular Python unit testing library, \verb|unittest|, does not do a collection pass),
we measure two different and related execution times:

\begin{itemize}
      \item The real unit test runtime, which is how long it takes for the unit test to run, including the collection
            time. We call this $T_{\text{real}}$. We collect this in the same way as the system test run, using the Python performance counter,

      \item The unit test execution time, which is how long it takes the execution pass of PyTest. We measure this using the open source library
            \href{https://pypi.org/project/pytest-execution-timer/}{pytest-execution-timer} \cite{pytest-exec-timer}. We call this $T_{\text{exec}}$.
\end{itemize}

Recall we have two categories of function-under-test, a non-buggy and a buggy one. We measure speedups of running the test
on both non-buggy and buggy version of the codebases we have, to capture two different experiences when using the
tests that ExploTest generates.

We generate tests only on the non-buggy version, for two reasons. First, an oracle carved from known-good code is
correct by construction; one carved from the buggy version would encode buggy behaviour.
Second, this matches the standard regression-test direction: write the test against correct code,
then replay it against the faulty version.

Having considered all combinations of buggy, non-buggy code and execution vs. real unit test time, we will measure 2 metrics presented
below to answer RQ3 on both non-buggy and buggy code to cover all use cases of the tool:

\begin{itemize}
      \item The real speedup $T_\text{sys}/T_\text{real}$, which is the ratio of the system test execution time and the total unit test execution time
      \item The execution speedup $T_\text{sys}/T_\text{exec}$, which is the ratio of the system test execution time and the execution time of the core unit test code (i.e., the total unit test execution time, minus the test collection time)
\end{itemize}

\textbf{Carving overhead}

Another side of the unit test generation story is the overhead the tool has when creating unit tests.

While unit tests can speed up the testing process, if generating them takes too long, then the value of the tool is diminished.

To calculate carving overhead, we look at the system test run time when carving, and when not carving. We devise the metric $T_\text{sys with carving} / T_\text{sys without carving}$ to measure the carving overhead.
A ratio $> 1$ indicates a slowdown.
