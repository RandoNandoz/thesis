\section{Answers to RQs}

\subsection{RQ1: Test quality}

In RQ1, we posed: \textit{To what extent does the unit test carving tool we created generate viable
    unit tests when applied to popular,
    open-source Python repositories?}


\hyperref[sec:RQ1Results]{Table C.1} shows ExploTest is able to generate at least 1 test for 16 curated bugs out of 22 bugs
from the software repositories we choose. We consider these runs successful. $72\%$ of ExploTest runs were able to create
some kind of syntactically valid unit test. The 6 unsuccessful runs are examined further in RQ4.

Out of the 16 runs that produced a unit test, all were able to correctly set up the arguments to the function
and successfully called the function.
$72\%$ of ExploTest runs create unit tests that when run, reach the function-under-test.

Manual analysis of the returned object's state showed that 14 out of 16 of the functions-under-test exercised by the unit test
had the same output as the function-under-test being exercised by the system test. The 2 cases of incorrect function object
were due to the custom serialization protocol a parameter used; ExploTest has no control over the customized serialization code
that implementors use.
$64\%$ of ExploTest runs create unit tests that reach and correctly call the function-under-test the same as a system test would.

AutoAssert was able to create assertions for 12 out of 14 of the tests that correctly call the functions-under-test.
$55\%$ of ExploTest runs create unit tests that correctly exercise the function-under-test and have an oracle.

11 out of 12 tests with assertions when run on correct code pass, and only 2 tests fail when run on incorrect code.
Only $9\%$ of the tests generated by ExploTest actually expose the bugs they intend to detect. Our approach
to generating assertions is likely too naive or conservative. Further discussion on improving the quality of
assertions will follow in section 5.

\subsubsection{RQ1 Answer}

\noindent\fbox{%
    \parbox{\textwidth}{%
        64\% of ExploTest runs create unit tests that correctly exercise the behaviour of their carving targets.
        Only 9\% of runs create tests that expose buggy behaviour. Test quality degrades sharply at the assertion generation
        phase: only 2 of 14 correctly-excerising tests produce assertions that expose the target bug.
    }%
}


\subsection{RQ2: Unit test viability by software attributes and family}

RQ2 poses the question: \textit{What software attributes and family are associated with the differing viabilities
    of unit tests created by the tool?}

\subsubsection{What attributes contribute to ExploTest's ability to create a valid unit test from a carve target and an integration test?}

First, let us define what a valid unit test is: a valid unit test created by ExploTest is a unit test that successfully sets up the
arguments for its carve target (i.e., the function-under-test), and calls the function in such a way where the function call
is equivalent to one in the system test. Notably, the tested created by the tool need not an oracle, as for exploratory uses
of ExploTest, this is sufficient; the issue of oracle creation is pervasive with automatic unit test generation tools \cite{fraser2015does},
which this tool does not aim to address.

Looking at our tests and function-under-tests, we can gather some patterns. Particularly, we'll study the two tests
that are able to fully catch bugs, however these patterns are consistent throughout all our successful runs.


Observe the test generated when run on the \verb|update_headers| procedure from \verb|httpie|
in \hyperref[sec:httpie-fully-functional-ut]{A.1}. The test is able to detect the bug since in
the buggy version the check for \verb|value is None| is not present and causes the reference
to \verb|value.decode| to throw a \verb|NameError|.

The other test for in the ``uncovers bug'' category for the \verb|sanic| web framework in \hyperref[sec:sanic-fully-functional-ut]{A.2}
has similar attributes to the first. The function-under-test, \verb|url_for|'s only external data dependencies are on the \verb|re|
regular expression library and the \verb|urlencode|, \verb|urldecode| functions from \verb|urllib| library which do not require
any previous state to be set up for them.

\begin{itemize}
    \item The function-under-test has limited external dependencies for it to function.
    \item The function-under-test's external dependencies do not require their states to be set; they
          are stateless, or the default state is sufficient.

    \item Target function-under-tests have serializable parameters. This is unsurprising, considering
          the tool uses record-and-replay techniques to carve its tests.
    \item Carve candidates have clearly defined entry points, and are reachable in the call graph.
          % \item Every type used in the function-under-test is serializable. The \verb|request_headers|
          %       parameter is a dictionary of strings and bytes, which are both serializable.
\end{itemize}


\subsubsection{What attributes contribute to ExploTest's ability to create oracles that are able to detect bugs?}

Previously, we defined a valid unit test is and we explicitly excluded oracle generation from our definition of validity.
However, it is still worth looking at our naive oracle generation process, since verifying software behaviour is an important
part of a unit test suite.

The story is similar with oracles---serializable and simple return values mean ExploTest can create meaningful oracles
that capture defects.
From our benchmarks, the only two runs that successful capture defects are from \hyperref[sec:httpie-fully-functional-ut]{httpie-3}
and \hyperref[sec:sanic-fully-functional-ut]{sanic-3}.

Namely, the httpie-3 run has a crash oracle when accessing an invalid property, which immediately fails the test.

In the sanic-3 run case, the function-under-test returns a string, which is easily serializable and comparable using our existing tools.

\subsubsection{Viability by software family}

We aggregate the ratio of successful tests by software category in \hyperref[sec:RQ2Results]{Table D.1}. The Data visualization,
Command line utility, and Web framework categories are discarded since there is not enough data to draw a meaningful analysis.

Most categories are within the 64\% range. Notably, source code analysis tools work well with ExploTest; we suspect source code
analysis tools work well as they do not interface with native code, but instead operate on simple, tree like structures. Other results
are not notable enough to discuss.



\subsubsection{RQ2 Answer}

\noindent\fbox{%
    \parbox{\textwidth}{%
        We define two different viabilities for the unit tests generated by the tool: (1) their ability to exercise the function-under-test,
        and (2) their ability to generate an oracle that successfully detects software defects.

        \medskip

        We find 4 major software qualities that contribute
        to (1): limited external dependencies (the absence of global state), and if there is a dependency on external state, no external setup
        is required for them. Target function-under-tests must have serializable parameters, since the tool is a record-and-replay tool.
        The function must be discoverable by the test, namely, they cannot be closures and must be directly exposed to other functions.

        \medskip

        Limited data is present for (2), however, the story is similar to (1)---simple function outputs that are serializable and easily comparable
        mean oracles can be easily generated

        \medskip

        In terms of software, we categorize our bug origins from 6 different software families, however, we have limited data to make a statistically
        significant conclusion. With our limited data, we find that bugs from source code analysis tools like black and pylint are associated
        with more runs that generate valid unit tests. We hypothesize that this is due to these tools having many qualities from (1), since
        source code is easily representable with serializable objects. This furthers strengths the validity of our answer for (1).
    }%
}




\subsection{RQ3: Speedup over integration tests}

For RQ3, we asked ``What is the amount of speedup offered by the unit tests created by the tool
compared to their system test runs?''.

The appendix section \hyperref[sec:speedup-data]{E.1} contains the data desired in section \hyperref[sec:answering-rq3]{3.3.3}.
An analysis of the data follows to answer RQ3.

\subsubsection{Real speedups}
Let us first discuss the real speedups for both non-buggy and buggy cases.

Speedups are similar between buggy and non-buggy cases, and differences can be attributed
to different code paths being executed in the buggy and non-buggy cases. We discuss the non-buggy case here. The data for the
buggy case can be found in the appendix, in section \hyperref[sec:speedup-data]{E.1}.
Interesting differences between the non-buggy and buggy cases will also be discussed in this section.

Real speedups are limited for the non-buggy and buggy cases. The median real speedup over both non-buggy and buggy cases offered by the tool is
$0.9 \times$, which is actually a slowdown.

Only 5 test suites generated (pylint-10402, sanic-1, sanic-3, tornado-15, and tornado-4)
by the tool offer a statistically significant speedup. In these cases, we find the collection time significantly dominates the execution
time of the tests. Furthermore, the collection time must also be greater than the system test run time. This makes sense, since
many of our system tests are minimum reproducible examples from bug reports, which do not exercise a significant amount of the system.

This invites the question: is there a relationship between real speedup and the system test time?

To answer the question, we perform a simple linear regression, shown in Figure~\ref{fig:real_speedup_plot}.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=\linewidth]{plots/plots_regression_real.png}
    \caption{Regression analysis for mean system test time and speedup}
    \label{fig:real_speedup_plot}
\end{figure}

We do observe a statistically significant and strong linear relationship between the system test time and real speedup ($R^2 = 0.849, p < 0.001$).

This is consistent with our understanding of the underlying benchmark suite, namely, tornado and sanic, the two code bases that exhibit
a speedup are both web frameworks, and their underlying system test benchmark involves \verb|curl|ing an endpoint, which imposes
a significant overhead due to network requests having extra overhead through the operating system space network stack.

From this analysis, we can infer that ExploTest, and other similar unit test carving tools only create a meaningful speedup against system test
when the system test are already quite expensive.

\subsubsection{Execution speedups}
Since our arbitrary choice of unit test library imposes a significant overhead when in its collection pass, we invent a new metric,
the execution speedup, which is the total unit test execution time minus the collection time. One can think of this as the speedup
we would have had if we had chosen to use the stock Python \verb|unittest| library instead of the PyTest library, since the \verb|unittest|
library does not perform an expensive collection pass by default. Earlier versions of ExploTest targeted the \verb|unittest| library
instead of PyTest, and an analysis by Zihao Huang, the original creator of the tool found similar speedups.

We see a median speedup of $32.18 \times$---in the time that it takes one system test to execute its code, the median unit test suite
generated by ExploTest can run $32.18$ times.

In Figure~\ref{fig:exec_speedup_plot}, we perform a similar regression analysis for the speedup in the non-buggy case. The relationship is similar with
the buggy case, and can be found in the appendix.

\begin{figure}[htbp]
    \centering
    \includegraphics[width=\linewidth]{plots/plots_regression_exec.png}
    \caption{Regression analysis for mean system test time and execution speedup}
    \label{fig:exec_speedup_plot}
\end{figure}

Unlike in the real speedup case, we find no relationship ($R^2 = 0.185, p = 0.110$) between execution speedup and system test time.
This is consistent with our understanding of unit testing: unit tests exercise less code
than system tests, so their execution time is largely independent of the system test's duration. If we treat system test execution time as a proxy for the amount of code exercised by the system test, and unit test execution time as a proxy for the amount of code exercised by the unit test, the lack of correlation follows naturally.

\subsubsection{Real and execution speedups: Non-buggy vs buggy}

We observe the distributions of the speedups between non-buggy and buggy runs of the test suites generated by the tools to
compare their speedups in Figure~\ref{fig:quad-distribution-buggy-nonbuggy-analysis}.

\begin{figure}[htbp]
    % \centering
    % \includegraphics[width=\linewidth]{plots/plots_merged_speedup_boxplot.png}
    \centering
    \includegraphics[width=\linewidth]{plots/plots_real_speedup_boxplot.png}

The distribution of speedups is similar across the board. However, notably, the execution speedup
for black-12 is $10 \times$ worse in the buggy case. The root cause for this is that the buggy case
of the repository is significantly faster, as it does not perform a full traversal of the AST, while the
test suite (which contains 400+ tests) replays the full traversal.

\subsubsection{Carving overhead}
To properly evaluate the speedup, we should also consider the cost of carving the test. This RQ does not
discuss finding the correct place to mark the carve target with \verb|@explore|, instead, it looks at the performance
penalty of running the system test with carving, since a significant slowdown of the system test when carving
significantly impedes the usability and cost-benefit of the tool.

In appendix section E.2, we collected the median runner time with carving vs without carving. We found $10/15$ of the tested
repositories showed a statistically significant difference between the two runs ($p < 0.05$), with an average slowdown (geometric mean)
of $1.43\times$.



\subsubsection{RQ3 Answer}
\noindent\fbox{%
    \parbox{\textwidth}{%
        Comparing speedup requires assessing many facets. The creators of the tool made an arbitrary choice of using PyTest as
        the target test framework for unit tests. PyTest has a collection pass that collects metadata associated with unit testing (e.g., config files)
        which may dominate the perceived runtime of the tests. We devise two metrics: real speedup, which is the perceived runtime when running the
        test suite, and execution speedup, which is the runtime of only the test code (i.e., the perceived runtime minus the time it takes for the collection
        pass to run). Another facet of performance is the performance overhead of the tool. If the system test slows down significantly, that must
        impede the tool's usability.

        \medskip

        A real speedup is offered for 5/15 of the repositories that generated a valid unit test. The median real speedup was $0.9 \times$, which was a slowdown.
        We find a positive relationship with system test runtime and real speedup ($R^2 = 0.849, p < 0.001$). In terms of execution speedup, the suites generated by
        ExploTest had a median speedup of $32.18 \times$. There is no statistically significant relationship between execution speedup and system test runtime ($R^2 = 0.185, p = 0.110$).
        Looking to performance overhead, we measured the system tests with and without the carving tool attached. We observed a statistically significant slowdown in 10/15 cases.
        The average (geometric mean) slowdown was $1.43\times$.

        \medskip

        This suggests ExploTest is well suited for use when system test times are significant, drawing from the relationship between real speedup and system test time.
        Discounting the unit test framework overhead however, carved unit tests created by the tool offer a significant speedup. There is overhead to using the tool in 10/15 cases,
        however. The average overhead was a $1.43 \times$ slowdown.
    }%
}


\subsection{RQ4: Causes of tool being unable to generate unit tests}
RQ4 asked: \textit{What modes of failure occur when the tool fails to create any kind of unit test?}

To answer RQ4, we perform a root-cause-analysis as to why the tool fails to create a unit test. We present the results in Table~\ref{tab:rq4-failures}

\begin{table}[ht]
    \centering
    \caption{Root causes of ExploTest test generation failures (6 of 22 bugs)}
    \label{tab:rq4-failures}
    % \footnotesize
    \begin{tabular}{llll}
        \toprule
        \textbf{Bug}  & \textbf{Decorated function} & \textbf{Unpicklable object(s)}   & \textbf{Object type}              \\
        \midrule
        sanic-2       & \texttt{serve}              & \texttt{loop}, \texttt{sock}     & Event loop, OS socket             \\
        youtube-dl-12 & \texttt{extract\_info}      & \texttt{self}                    & I/O streams (\texttt{sys.stdout}) \\
        tornado-10    & \texttt{write\_message}     & \texttt{self}                    & \texttt{IOStream} (OS socket)     \\
        gunicorn-3144 & \texttt{\_\_init\_\_}       & \texttt{self}, \texttt{unreader} & Socket-backed I/O                 \\
        ipython-14930 & \texttt{shebang}            & \texttt{self}, \texttt{stream}   & ZMQ kernel, subprocess pipe       \\
        \midrule
        seaborn-3553  & \texttt{boxplot}            & (pickling succeeds)              & \texttt{.mT} crash                \\
        \bottomrule
    \end{tabular}
\end{table}

5/6 failures share a single root cause. The \verb|self| parameter of an instance method contains OS-level resources (sockets, file descriptors, event loops)
that dill cannot serialize, and are ephemeral resources. The last error occurs in the assertion generation phase. The assertion generator tries to read
\verb|.T| for a \verb|np.Series|, which consequently calls \verb|.mT| (matrix transpose) in NumPy, however, since an \verb|np.Series| is a 1 dimensioned object,
an error is thrown, causing the tool to crash.

This suggests a significant impediment to ExploTest and other unit test carving tools is the object creation problem. If one wants to re-create these ephemeral
OS-level resources, there must exist some degree of action-based carving required to re-request the resources from the OS.

\subsubsection{RQ4 Answer}
\noindent\fbox{%
    \parbox{\textwidth}{%
        In 5/6 cases, ExploTest is unable to generate a unit test suite due to unpicklable arguments. All unpicklable arguments in our examples are from
        OS-level resources. The single case that causes the tool crash is from an attribute access in the assertion generation stage, which calls a function
        that throws an uncaught exception.
    }%
}

\subsection{RQ5: Tool usage}
RQ5 poses: \textit{What is the procedure to use ExploTest to catch defects in software?}

\subsubsection{Author's technique}
We discuss the technique the author used to carve the tests for QuAC first.

\begin{enumerate}
    \item Read the bug report; identify the broken feature
    \item Set up the system test to expose the bug
    \item Search the codebase for that feature is implemented
          \begin{itemize}
              \item The first strategy the author tries is searching the code for strings related to the feature, to try to localize the location.
              \item If this strategy is unable to locate the target function, the author moves on to dynamic analysis methods, which include:
                    \begin{itemize}
                        \item \texttt{printf} debugging, trying to localize the buggy function by looking at the lines reached by the function
                        \item Attaching the Python debugger in Visual Studio Code to locate the target function
                        \item Using code-coverage tools like coverage.py to understand the control-flow of the program
                    \end{itemize}
          \end{itemize}
    \item Place an \verb|@explore| statement on the target functions, or target functions, and running the system test to carve the unit test
    \item If ExploTest succeeds, then we have a regression test, otherwise, document accordingly
    \item If no carve candidate can be found, document as a failure mode (for discussion in RQ4)



\end{enumerate}


\subsubsection{Case Study}

Here, we present a case study of how one would use ExploTest in a manner to help in fixing a bug. The bug is \verb|ipython-14930|, where
the shell shebang, an escape sequence from the IPython shell to call OS commands does not handle streaming multi-byte, unicode characters properly.
The bug report is in Figure~\ref{fig:ipython-bug-report}.

\begin{enumerate}
    \item Reading the bug report, the author internalizes the bug and how to reproduce it.
    \item The author reproduces the bug interactively to verify it is actually present.
    \item The keywords ``streaming'', and the usage of \verb|%%| are the first keywords that jump out to the author.
    \item The author searches for \verb|%%|, which produces a number of results,
          many of which are false positives, and then the author searches for the keyword \verb|streaming| in
          the codebase, which returns few meaningful results, then finally \verb|stream|. The author quickly
          glances at the options and identifies the \verb|shebang| method as a candidate, as the comments
          (i.e., source code level documentation) explains the behaviour, and it matches our desired behaviour.
    \item The streaming code is inside a closure, handled by \verb|_handle_stream|, so the author must
          place the \verb|@explore| on \verb|shebang|, the parent method.
    \item The author then runs the system test,
          interactively with the \verb|ipython| command to carve a test. The generation fails, and
          a discussion is followed in RQ4.

\end{enumerate}
